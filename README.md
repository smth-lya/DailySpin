## DailySpin 🎡 - рандомайзер задач, который сделает выбор за тебя

### О проекте

**Проблема.** Планирование - это когнитивная нагрузка. Нужно расставить приоритеты, выбрать первую задачу, не прокрастинировать.

**Решение.** Убрать выбор как этап. Пользователь собирает задачи в бэклог, расставляет приоритеты, а дальше решает система. Механика выбора - колесо фортуны.

> TL;DR: У каждой задачи есть "вес" - он зависит от приоритета и сложности. Чем вес больше, тем больше сектор на колесе и выше шанс выпадения. От пользователя требуется только расставить приоритеты.

**Функциональность:**
1. Колесо фортуны - случайный выбор задачи с учетом ее веса.
2. Доска задач - стикеры, дизайн которых зависит от сложности задачи.
3. Таймер "Непрерывной работы" без паузы.
4. Обратная связь - после работы пользователь отмечает статус: "Выполнено" или "Требуется больше времени". Система автоматически корректирует расписание или возвращает задачу в пул.

> Система должна покупать визуалом. Сложная задача - потенциально крутой стикер со зрелищной анимацией при ее закрытии.

## Архитектура. Техническая реализация

> Проект писался с ограничением: никаких сторонних фреймворков, если что-то и использовать, то только самописное.

Сначала запросы обрабатывались через `HttpListener` с кучей условий под каждый путь. Это быстро превратилось в кашу. Пришлось залезть в исходники ASP.NET Core и написать собственный минималистичный аналог.

### Что получилось собрать:
1. DI-контейнер - реализация внедрения зависимостей (писал, разбирая [это видео](https://www.youtube.com/watch?v=dCkYP03lXOs))
2. ORM-прослойка - разбирал реализацию на основе [этого видео](https://www.youtube.com/watch?v=UvO-8p3JqW0)
3. MiddlewarePipeline - взял идею с ASP.NET: запрос проходит по цепочке, обрастает контекстом, попадает в нужный endpoint
4. Атрибуты и маршрутизация - `[Route]`, `[Authorize]`, `[HttpGet]`, `[HttpPost]`, `[FromBody]`, `[FromQuery]`, `[FromHeader]`. Контроллеры получают зависимости из DI, аргументы - из запроса
5. JWT-авторизация - access + refresh токены.
6. Архитектура - пытался соблюдать DDD-расслоение: Domain, Application, Infrastructure, API
7. Endpoints - регистрация, вход/выход, валидация токенов, базовый контроллер колеса. PostgreSQL подключение, репозитории.

### Что не было доделано
1. Бизнес-логика - с ней провозился меньше всего. Основное время ушло на фреймворк.
2. Фронтенд - заглушка. В текущей версии проекта интерфейс не реализован.

### Итог

Проект стал исследованием внутреннего устройства веб-фреймворков. В процессе удалось детально разобраться с механизмами ASP.NET Core, DI и ORM через написание собственной реализации.
